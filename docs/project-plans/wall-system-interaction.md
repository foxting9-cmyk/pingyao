# 城墙体系交互优化方案

本文档用于说明：首页中“城墙体系 / 古城总览”这一块，如何从当前的基础版本优化成更接近参考图的效果，以及如何实现“点击某个构件后，再出现详细介绍”的交互。

---

## 一、结论先说

可以实现，而且建议做成：

1. **主画面总览图 + 热点标注**
2. **左侧构件列表联动右侧图面**
3. **点击热点或列表项后，右侧或底部弹出详细介绍卡**
4. **移动端改成抽屉层或底部浮层**

也就是说，参考图里那种“总览图上有多个信息点，点开后看到更细内容”的效果，在你这个项目里完全可以落地，而且不一定非要一开始就上很重的 3D。  
**最推荐的正式版路线是：先做 2D 总览图 + 伪 3D 标注 + 详情卡联动，再逐步接入 GLB/three.js。**

---

## 二、为什么参考图看起来好

参考图的好看，不只是因为它“画得细”，更重要的是它做对了这几件事：

1. **一张总图先建立全局认知**
   用户第一眼就能明白：这是一座完整的城，不是零散的建筑截图。

2. **信息点不是同时爆出来**
   重点信息被拆成“可点、可看、可展开”的形式，所以画面虽然复杂，但不乱。

3. **图像和说明是联动的**
   左边讲“城门 / 角楼 / 瓮城”，右边总图能对应找到位置，认知很顺。

4. **层次明确**
   主体是古城总览，其次是红色标注点，再其次才是边缘解释和附图。

你现在要优化的方向，不是简单“把图换得更复杂”，而是要做成：

**先看懂整体，再点击理解局部，再进入详细资料。**

## 三、推荐的最终效果结构

建议把这块做成一个完整的交互模块，布局如下：

### 1. 左侧：构件列表

展示：

- 城门
- 瓮城
- 角楼
- 马面
- 敌楼
- 垛口

每一项默认展示：

- 名称
- 数量
- 一句简述

点击后：

- 当前项高亮
- 右侧总览图对应热点高亮
- 下方或右侧出现详细介绍

### 2. 右侧：古城总览图 / 伪 3D 图 / GLB 模型

这一块可以有 3 个演进层级：

#### 方案 A：2D 总览图

用你发的这类“古城总览图”作为主视觉底图，在上面覆盖热点点位。

优点：

- 开发成本最低
- 效果最稳定
- 最接近参考图

#### 方案 B：2D 图 + 伪 3D 浮层

底图仍然是总览图，但热点点击后，在某个局部区域弹出小卡片、构件插图、局部放大框，看起来像“图上又长出一层信息”。

优点：

- 比纯列表更有设计感
- 不会像纯 3D 那样重

#### 方案 C：three.js / GLB 模型视图

在右侧放 GLB 模型，让用户旋转查看，热点挂在模型附近。

优点：

- 交互感强
- 更容易体现“空间结构”

缺点：

- 性能压力更大
- 首屏体积会明显增加
- 标注点与模型坐标要仔细调

**推荐顺序：A -> B -> C。**

也就是：

先把“参考图式的信息逻辑”做对，再逐步接 3D。

## 四、“点击后出现详细介绍”怎么做

答案是：**完全可以，而且非常适合这个项目。**

建议做成 3 层交互：

### 第 1 层：轻提示

用户鼠标移到热点上时：

- 点位发光 / 放大
- 显示小标签，如“城门”“角楼”

这一步是告诉用户：这里可以点。

### 第 2 层：展开简介

点击热点或左侧列表项后：

- 右侧出现一个中等大小的信息卡
- 展示：
  - 名称
  - 数量
  - 所在位置
  - 一句话功能说明
  - 一张局部图 / 示意图

这是“先展开一层”的效果，最接近你说的“点击就再出现一个详细介绍”。

### 第 3 层：查看完整资料

在简介卡里再放一个按钮：

- `查看详情`
- `查看证据图`
- `查看关联关系`

点击后再出现：

- 抽屉层
- 侧边详情面板
- 模态框

里面可放：

- 详细文字说明
- 构件位置图
- 相关史料来源
- 与其他构件的关系

这样就形成：

**总览 -> 点击 -> 简介卡 -> 再点 -> 完整详情**

这个层级非常适合知识图谱型项目。

## 五、推荐的交互设计

### 桌面端

推荐布局：

- 左：构件列表
- 右：总览图 / 模型
- 右下或右侧：详情卡

交互逻辑：

1. hover 热点：小提示
2. click 热点：详情卡展开
3. click 详情卡按钮：打开完整侧边栏

### 移动端

不要照搬桌面端。

推荐改成：

1. 上方：总览图
2. 下方：构件列表
3. 点击热点后：从底部弹出抽屉层

原因：

- 手机屏幕窄
- 侧边栏体验差
- 抽屉层更符合移动端习惯

## 六、推荐的数据结构

建议给每个构件准备统一结构：

```json
{
  "id": "gate",
  "name": "城门",
  "count": 6,
  "positionLabel": "南门 / 北门 / 东门 / 西门",
  "summary": "城门是古城防御与交通转换的关键节点。",
  "detail": "这里放更完整的说明文字，可包含历史背景、结构功能与资料出处。",
  "image": "/images/wall/gate-detail.png",
  "hotspots": [
    { "x": 32, "y": 68, "label": "南门" },
    { "x": 50, "y": 14, "label": "北门" }
  ],
  "sources": [
    "《平遥县志》",
    "平遥古城保护研究资料"
  ]
}
```

这样未来无论你用：

- 2D 总览图
- SVG 标注
- Canvas 标注
- GLB 模型热点

都可以共用一套数据。

## 七、组件拆分建议

建议拆成这些组件：

### 1. `WallSystemSection.jsx`

整个模块容器，负责管理当前选中的构件。

### 2. `WallStructureList.jsx`

左侧构件列表。

职责：

- 渲染列表
- 高亮当前项
- 点击切换当前构件

### 3. `WallOverviewBoard.jsx`

右侧总览图或模型区。

职责：

- 显示 2D 图 / 3D 模型
- 渲染热点
- 处理热点点击

### 4. `WallHotspot.jsx`

单个热点组件。

职责：

- hover 效果
- click 事件
- 标签浮层

### 5. `WallDetailCard.jsx`

构件详情卡。

职责：

- 展示摘要
- 展示局部图
- 提供“查看详情”入口

### 6. `WallDetailDrawer.jsx`

完整资料层。

职责：

- 展示大段说明
- 展示资料来源
- 展示关联构件

## 八、如果接 three.js，怎么实现“点击出现详细介绍”

如果你要把 `GLB` 真正做成可点击的交互模型，有两种主流方式：

### 方式 A：模型外部叠加 HTML 热点

做法：

- 模型在 Canvas 中展示
- 用 `Html` 或者屏幕坐标换算，把热点标签叠在模型附近

优点：

- 好做
- 文字样式自由
- 更容易做详情卡

缺点：

- 点位要手动调位置

### 方式 B：模型网格点击

做法：

- 给 GLB 里不同网格命名
- 通过 three.js 点击射线检测具体 mesh
- 根据 mesh 名称打开详情

优点：

- 更“真 3D”

缺点：

- 模型结构必须干净
- 如果 GLB 没有合理拆 mesh，会很难点

**推荐你当前项目优先用方式 A。**

也就是：

先把模型当作展示底座，再叠热点与信息卡，而不是一开始就追求“点模型本体的某个三角面”。

你现在不要急着一上来就把所有东西做成“纯 3D 复杂交互”，最优路线是：

这样效果最好、风险最小、也最符合你这个首页叙事项目的节奏。
